Javascript Is:

  • Dynamic as opposed to Static
  • Loosely typed as opposed to Strictly typed
  • Weakly typed as opposed to Strongly typed
  • Prototypical as opposed to Classical
  • Functionally Scoped as opposed to Lexically Scoped
  • High Level as opposed to Low Level
  • Object Oriented with a dash of Functional

BUT WHAT DOES IT MEAN?

Dynamic

  • Dynamic means the code does not need to be pre-compiled to run
  • Many tasks that other languages do during compilation are done during runtime
  • Dynamic languages are easier to work with, but more error prone

Loose

  • Dynamic languages are loosely typed: type is figured out at runtime
  • Variables, Functions, etc do not need to specify a type
  • JS has types! (we have typeof for a reason). The loosely typed system and lack of annotations just hides these details from you. I think this is a bad thing, personally
  • JS primitive types are: Boolean, Null, Undefined, Number, String, Symbol (ES6 only)
  • And Object
  • Arrays are not a type in JS, technically. They are just objects.

Weak

  • Types can be changed on a whim, via casting or coercion
  • JS won't care if you change your int 5 to a float 5.5, they're all numbers to js
  • You can even changes numbers to strings and vice versa, and the only error you'll get is if you try to use the wrong method on them. A tool like flow, or just basic type checking, can prevent this problem
  • You've done type coercion if you've ever used a number or string method

Prototypical

  • The big one, the most important, and the most misunderstood
  • Most languages are Class based, javascript is not
  • This is a good thing, but unfortunately the implementation of Prototypes in JS is not exactly user friendly
  • Every object in JS is part of a prototype chain, and has a prototype
  • Prototypes are just Objects (see a theme yet)
  • 2 Types of Prototypes: Delegates and Mixins
  • Delegates persist across all instances of a thing, Mixins are just copies
  • When you create a new Array, it inherits the Delegate prototype of the Array object, gaining access to all Array methods in the process

Functionally Scoped

  • Most languages are block scoped, not function scoped
  • In JS only Functions can create scope, if, else, etc do not
  • This will change in ES6, JS will also have Lexical scope as well as Function scope
  • You can actually see scope, just use console.dir on a function

High Level

  • The technical details of the computer are abstracted away from you
  • Basically, you don't have to manage memory, or interact directly with hardware, etc
  • By nature all Dynamic languages are high level
  • High level languages: Javascript, Ruby, Python, PHP, Go
  • Low level languages: C, Lisp, Rust, Nim, Swift (sorta)

Object Oriented

  • Everything except for type primitives in JS is an object. EVERYTHING ( except undefined )
  • JS coerces number, string, boolean primitives to Objects. undefined is the exception
  • Numbers? Objects. Functions? Objects. Arrays? Objects Strings? Objects
  • Think back to the slide on prototypical inheritance
  • This is why myNum.toString() works

Scope, Closure, and Context

  • Scope: Is created by a Function (in ES5, in ES6 JS is also lexical)
  • Closure:
    • Is when you have scope within scope (ie, functions within functions)
    • Closures inherit the scope of their parent function, allowing you to share variables between scopes
    • Using Closures to create privileged and private variables/functions is a very common pattern
  • Context: Is the value of the this keyword, which i'm gonna need another slide to explain

This

  • This behaves weirdly in javascript compared to other languages
  • By default it refers to the window or global object, and with 'use strict' on it returns undefined
  • Other than that, it's value depends on where and how it's used
  • This called in a function refers to where the function was called, not the internal scope of the function
  • Yet when used inside an object literal it works as expected, referring to the object
  • There are so many exceptions and rules i'll just refer you to the MDN page on it

Callbacks and Lambdas

  • Javascript supports Closures, Functions as Objects, Higher Order Functions and Lambdas
  • aka First Class Functions, or Functional Programming, depending on who you talk to
  • Higher Order Functions are Functions that can take Functions as parameters
  • Lambda Expressions is just when you can use a function's return as a value

So How Do Callbacks work anyway?

  • I'm glad you asked Little Jimmy
  • A callback is just a function, used as a parameter in another function
  • It's basically just a pattern, often used to handle the result of some previous action
  • Promises will prolly replace Callbacks for async, but callbacks will always have their uses

Prototypical Inheritance

  • From Wiki:
  • Prototype-based programming is a style of object-oriented programming in which behaviour reuse (known as inheritance) is performed via a process of cloning existing objects that serve as prototypes. This model can also be known as prototypal, prototype-oriented, classless, or instance-based programming.
  • You can create a Class system with Prototypes. You cannot create Prototypes with Classes.
  • You use it all the time, whether you realize it or not.

Prototypical Inheritance 2

  • A Prototype then is just a re-usable object available to all instances of something.
  • If you create a new jQuery object, you have access to all of jQuery's methods.
  • If you create a new Array, you have access to all Array methods ( slice, forEach, etc )
  • If you create a new HTMLElement, you have access to the DOM methods ( .getElementBy etc etc )
  • This kind of Prototype is called a Delegate Prototype

Prototypical Inheritance 3

  • Everything in Javascript descends from the Object prototype (its Objects all the way down)
  • All prototypes inherit the methods and properties of the prototype above them
  • This is called the Prototype Chain, the example below is a DIV from the DOM api
  • Object -> EventTarget -> Node -> Element -> HTMLElement -> HTMLDivElement
  • Altering the prototypes of existing Objects is considered bad practice
  • If we changed what Array.slice() did for instance, we could potentially break everything in the app that relied on Array.slice()
  • Despite what I literally just typed above: altering the prototype like this is what a lot of polyfills are actually doing, although smart ones fallback to the native if available

Prototypical Inheritance 4

  • For one off inheritance, you can also just straight up copy objects
  • This approach is called a Mixin
  • When you use a library's extend function, this is usually what they're doing
  • You can combine the two techniques, or just copy from many objects if you want
  • What happens if you copy an object and the keys conflict with another object's? Shit breaks, sorry
  • This is easily avoided with proper scoping though
  • A Prototypical Inheritance library like Stampit makes all the above very very easy
